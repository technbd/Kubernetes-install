
## Kubernetes Pods: 

A Pod is the smallest deployable unit in Kubernetes. A Pod can contain one or more containers that share:

- Network (same IP address)
- Storage volumes
- Namespace


Most applications use **one container per Pod**, but multiple containers can be used when they need to work closely together and are scheduled together on the same node.


### Pod Lifecycle:

```
Pending
   ↓
ContainerCreating
   ↓
Running
   ↓
Succeeded / Failed
```


### Common Kubernetes Objects and Their API Versions: 


_When creating Kubernetes objects using YAML manifests, there are **four mandatory** top-level fields:_
```
apiVersion: Which API to use?
kind:       What object to create?
metadata:   What is its identity?
spec:       How should it run?
```

_Kubernetes Object Manifest Structure:_
```
apiVersion:
Specifies which Kubernetes API version should be used to create the object.
Examples:
    - v1                          # Pod, Service, ConfigMap, Secret, PVC, PV
    - apps/v1                     # Deployment, ReplicaSet, StatefulSet, DaemonSet
    - batch/v1                    # Job, CronJob
    - networking.k8s.io/v1        # Ingress, NetworkPolicy
    - autoscaling/v2              # HorizontalPodAutoscaler (HPA)
    - rbac.authorization.k8s.io/v1 # Role, RoleBinding, ClusterRole, ClusterRoleBinding

kind:
Defines the type of Kubernetes object to create.
Examples:
    - Pod
    - Deployment
    - Service
    - StatefulSet
    - DaemonSet
    - Job
    - CronJob
    - Ingress

metadata:
Contains information that uniquely identifies the object.
Common fields:
    - name
    - namespace
    - labels
    - annotations
    - UID (generated by Kubernetes)

spec:
Defines the desired state and configuration of the object.
Examples:
    - Container images
    - Replica count
    - Ports
    - Volumes
    - Resource limits and requests
    - Selectors
    - Environment variables
```

### Kubernetes Controllers: 

| Controller | Purpose | Example Use Case |
|------------|---------|------------------|
| Deployment | Manages stateless applications, scaling, self-healing, and rolling updates | Web servers, REST APIs, frontend applications |
| StatefulSet | Manages stateful applications with stable network identity and persistent storage | MySQL, PostgreSQL, MongoDB, Kafka |
| DaemonSet | Ensures a Pod runs on every node (or selected nodes) | Log collectors, monitoring agents (Fluent Bit, Filebeat, Node Exporter) |
| Job | Runs a task to completion and then exits | Database migration, batch processing, data import/export |
| CronJob | Runs Jobs on a scheduled basis | Nightly backups, log cleanup, report generation |






### Kubernetes Pod Creation Methods: 

_Kubernetes resources (including Pods) can be created in two ways:_
- **Imperative** = quick commands (`kubectl run`, `kubectl create`)
- **Declarative** = YAML manifests (`kubectl apply -f` ) and is the preferred approach for real-world Kubernetes operations.


_Key Difference: `create` vs `apply`:_

| Command | Type | Behavior | 
| ------- | ---- | -------- | 
| `kubectl run ...` | Imperative | Direct command-line resource creation | 
| `kubectl create -f file.yaml` | Semi-declarative | Creates only; errors if resource already exists | 
| `kubectl apply -f file.yaml` | Declarative | Creates if missing, updates if it exists | 




Both commands create Kubernetes resources, but they work differently and are used in different scenarios.

_`kubectl create` vs `kubectl apply`:_

| Feature                   | `kubectl create`  | `kubectl apply`    |
| ------------------------- | ----------------- | ------------------ |
| Creates new resource      | ✅ Yes             | ✅ Yes              |
| Updates existing resource | ❌ No              | ✅ Yes              |
| Idempotent                | ❌ No              | ✅ Yes              |
| Best for                  | One-time creation | Ongoing management |
| Production use            | Less common       | Recommended        |
| GitOps friendly           | ❌ No              | ✅ Yes              |

## Pod Architecture Patterns (Types): 


### 1. Single-Container Pods (Most Common): 

One container per pod — used for most stateless apps. 

The "one-container-per-pod" model is the most common Kubernetes use case. In this model, the Pod is just a wrapper around a single container, and Kubernetes manages the Pod rather than the container directly.



#### Imperative Method: 

Create resources directly using `kubectl` commands.

**Advantages:**
- Fast and simple
- Useful for testing
- Good for troubleshooting


**Disadvantages:**
- Source of truth- None 
- Difficult to track changes
- Not suitable for production
- No version control




_Syntax:_
```
kubectl run <POD_NAME> --image=<image_name> [--port=port] [--replicas=replicas] [--env="key=value"] [--dry-run=client] [--overrides=inline-json] [--command] -- [COMMAND] [args...]
```



_Create a Pod directly:_
```
kubectl run nginx-pod --image=nginx:alpine

Or,
kubectl run nginx-pod --image=nginx:alpine --port=80

Or,
kubectl run nginx-pod --image=nginx:alpine --port=80 -o yaml
```




_Generate YAML from Imperative Command:_
```
kubectl run nginx-pod --image=nginx:alpine --port=80 --dry-run=client -o yaml
kubectl run nginx-pod --image=nginx:alpine --port=80 --dry-run=client -o yaml >> nginx-pod.yaml
```


_Apply Configuration:_
```
kubectl create -f nginx-pod.yaml

Or,
kubectl apply -f nginx-pod.yaml
```




#### Declarative Method: 

Resources are defined in YAML files and applied to the cluster.

**Advantages:**
- Version controlled (Git)
- Reproducible
- Easy collaboration
- Supports GitOps
- Production-ready

**Disadvantages:**
- Requires YAML files
- Slightly more setup




_Create a YAML File:_
```yaml
apiVersion: v1
kind: Pod      #Deployment #Service
metadata:
  name: nginx-pod

spec:
  containers:
  - name: nginx-pod     # containers section is array/dictionary 
    image: nginx:1.25
    ports:
    - containerPort: 80
```


_Apply Configuration:_
```
kubectl apply -f nginx-pod.yaml
```


_Delete Resource:_
```
kubectl delete -f nginx-pod.yaml
```





### 2. Multi-Container Pods (Advanced): 

Multiple containers run inside the same Pod and share networking and storage,  working together.


Use case:
- Log collection
- Monitoring agents
- Proxy containers


_Example-1:_
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: web-app

spec:
  containers:
  - name: nginx         # The Main Application Container
    image: nginx:1.25
    ports:
      - containerPort: 80
    volumeMounts:
    - name: logs
      mountPath: /var/log/nginx  # NGINX Writes logs to and stored in a shared volume

  - name: log-shipper      # The Sidecar Container
    image: fluentd:latest
    volumeMounts:
    - name: logs
      mountPath: /var/log/nginx   # Same folder as NGINX → so it can read logs in real time

  volumes:       # Shared storage volume for both containers
  - name: logs
    emptyDir: {}  # Deleted when Pod is removed
```



```
kubectl apply -f web-app.yaml
```


```
kubectl get pod
kubectl describe pod web-app
```


```
kubectl exec -it web-app -- bash
```


```
kubectl logs <pod-name> -c <container-name>

kubectl logs -f web-app -c nginx

kubectl logs -f web-app -c log-shipper
```

```
-- List Pods:

-- Detailed information:

```

## Types of Pods Based on Usage:


### 1. Standalone Pod: 

Created directly and not managed by any controller.

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: standalone-pod
spec:
  containers:
  - name: app
    image: nginx:1.25
```


### 2. Deployment Pods

Managed by a Deployment controller.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx

  template:
    metadata:
      labels:
        app: nginx

    spec:
      containers:
      - name: nginx
        image: nginx:latest
```


### 3. StatefulSet Pods

Used for databases and stateful applications. Each Pod gets a unique identity and storage.

```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: mysql
spec:
  serviceName: mysql
  replicas: 3
  selector:
    matchLabels:
      app: mysql

  template:
    metadata:
      labels:
        app: mysql

    spec:
      containers:
      - name: mysql
        image: mysql:8.4
```



### 4. DaemonSet Pods

Runs one Pod on every node.

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: node-exporter
spec:
  selector:
    matchLabels:
      app: node-exporter

  template:
    metadata:
      labels:
        app: node-exporter

    spec:
      containers:
      - name: node-exporter
        image: prom/node-exporter
```



### 5. Job Pods

Runs a task once and exits successfully.

```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: backup-job
spec:
  template:
    spec:
      containers:
      - name: backup
        image: busybox
        command:
        - sh
        - -c
        - "echo Backup completed"

      restartPolicy: Never
```


### 6. CronJob Pods

Runs Jobs on a schedule.

```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: backup-cronjob
spec:
  schedule: "0 2 * * *"

  jobTemplate:
    spec:
      template:
        spec:
          containers:
          - name: backup
            image: busybox
            command:
            - sh
            - -c
            - "echo Daily Backup"

          restartPolicy: Never
```
