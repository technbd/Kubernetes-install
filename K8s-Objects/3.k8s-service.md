## Kubernetes Services:

A Kubernetes Service is an abstraction that provides a stable network endpoint for a group of Pods.  Pods are ephemeral — they get created, destroyed, and rescheduled, and each gets a new IP address.

A Kubernetes Service solves this problem. It acts as an abstraction layer that defines a logical set of Pods and a reliable, permanent policy by which to access them (a stable IP address and DNS name).


### Why Services are Needed:

_Without a Service:_
- Pods receive dynamic IP addresses.
- If a Pod is recreated, its IP changes.
- Other applications cannot reliably connect to it.

_With a Service:_
- A stable virtual IP (ClusterIP) is assigned.
- Traffic is automatically load-balanced across matching Pods.
- Applications communicate using the Service name instead of Pod IPs.



### How it works:

A Service finds its Pods using **label selectors**. Any Pod with matching labels automatically becomes a backend for that Service — no manual registration needed.

```
Client → Service (stable IP/DNS)  → Pod 1
                                  → Pod 2
                                  → Pod 3
```



### Full Flow Example:

- `--port` : Service port (This is the port exposed inside the Kubernetes cluster.)
- `--target-port` : This is the actual port inside the Pod/container/application are listening on. If not specified: Kubernetes assumes- targetPort = port
- `nodePort` (External Port) : This is the port exposed on every worker node (external access). Door on every Kubernetes node. 
 

_Traffic flow:_
```
Client
  |
  v
NODE IP:30080   (nodePort)
  |
  v
Service:80      (port)
  |
  v
Pod:8080        (targetPort: container/Application)
```

## Types of Kubernetes Services:


| Type | Description |
| ---- | ----------- |
| ClusterIP (default) |  Exposes the Service internally, reachable only within the cluster |
| NodePort | Exposes the Service on a static port (typically between `30000-32767`) on every node's IP, making it reachable externally. |
| LoadBalancer | Provisions an external load balancer (typically via a cloud provider) |
| ExternalName | Maps the Service to an external DNS name, no proxying involved |




## ClusterIP (Default):

A ClusterIP service is the default type of service in Kubernetes. Used for internal communication for inter-cluster communication. Like web service & DB service. Don't allow external traffic.

Exposes the Service internally within the cluster.




### Imperative Method:

_Create Pod/Deployment:_
```
kubectl run nginx-pod --image=nginx:1.25
kubectl create deployment nginx-deployment --image=nginx:1.25 --replicas=3
```


_Create a ClusterIP Service directly:_
```
kubectl expose pod nginx-pod --type=ClusterIP --port=80 --target-port=80 --name=nginx-pod-svc

kubectl expose deployment nginx-deployment --type=ClusterIP --port=80 --target-port=80 --name=nginx-deployment-svc
```



_Generate YAML from Imperative Command:_
```
kubectl expose pod nginx-pod --type=ClusterIP --port=80 --target-port=80 --name=nginx-pod-svc --dry-run=client -o yaml

kubectl expose deployment nginx-deployment --type=ClusterIP --port=80 --target-port=80 --name=nginx-deployment-svc --dry-run=client -o yaml > nginx-deployment-svc.yaml
```


```
kubectl apply -f nginx-deployment-svc.yaml
```


_Verify directly on the host machine:_
```
kubectl get svc
curl http://CLUSTER-IP
```



_Verify inside a Kubernetes Pod:_
```
kubectl exec -it nginx-deployment-7c4cddbb89-djkgp -- bash

curl http://nginx-pod-svc
curl http://nginx-pod-svc.default.svc.cluster.local

curl http://nginx-deployment-svc
curl http://nginx-deployment-svc.default.svc.cluster.local
```




### Declarative Method:

Services don't track Pod IPs manually. Instead, they use **Labels and Selectors**. 

#### Example-1:

_Create a Pod YAML File:_
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
  labels:
    app: nginx-web       # This label links the Pod to the Service

spec:
  containers:
  - name: nginx-container
    image: nginx:1.25
    ports:
    - containerPort: 80  # Default Nginx port
```


_Create a Service YAML File:_
```yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx-pod-svc

spec:
  type: ClusterIP        # Change to NodePort if you need external access
  selector:
    app: nginx-web       # Must exactly match the Pod's label

  ports:
  - protocol: TCP
    port: 80             # The port the service exposes inside the cluster
    targetPort: 80       # The port the container is listening on
```



```
kubectl apply -f nginx-pod.yaml

kubectl apply -f nginx-pod-svc.yaml
```



#### Example-2:

Suppose you have a Deployment running 3 replicas of an nginx web app:

_Create a Deployment YAML File:_
```yaml 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
  labels:
    app: web-server

spec:
  replicas: 3          # Tells Kubernetes to run 3 identical copies of this Pod
  selector:
    matchLabels:
      app: web-server  # Tells the deployment which Pods to manage

  template:
    metadata:
      labels:
        app: web-server # This label must match the selector above

    spec:
      containers:
      - name: nginx-container
        image: nginx:1.25
        ports:
        - containerPort: 80 
        resources: {} 
```


_Create a Service YAML File:_
```yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx-deployment-svc

spec:
  type: ClusterIP      # Defines the service type NodePort or LoadBalancer
  selector:
    app: web-server    # Must match the template labels in the Deployment

  ports:
  - protocol: TCP
    port: 80           # Port the Service exposes
    targetPort: 80     # Port the containers are listening on
```


```
kubectl apply -f nginx-deployment.yaml

kubectl apply -f nginx-deployment-svc.yaml
```

## NodePort:

Exposes the Service on a static port on every cluster node.


### Imperative Method:


_Create a NodePort Service directly:_
```
kubectl expose pod nginx-pod --type=NodePort --port=80 --target-port=80 --name=nginx-pod-svc

kubectl expose deployment nginx-deployment --type=NodePort --port=80 --target-port=80 --name=nginx-deployment-svc
```



_Generate YAML from Imperative Command:_
```
kubectl expose pod nginx-pod --type=NodePort --port=80 --target-port=80 --name=nginx-pod-svc --dry-run=client -o yaml

kubectl expose deployment nginx-deployment --type=NodePort --port=80 --target-port=80 --name=nginx-deployment-svc --dry-run=client -o yaml > nginx-deployment-svc.yaml
```


```
kubectl apply -f nginx-deployment-svc.yaml
```


_Verify:_
```
http://NODE-IP:nodePort
```



### Declarative Method:

_Traffic Flow:_
```
Client
  |
  v
NodeIP:30080  (NodePort)
  |
  v
Service Port 80 
  |
  v
Pods (3 replicas)
  |
  v
Container Port 80 (targetPort)
```



#### Example-1:

_Create a Pod Service YAML File:_
```yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx-pod-svc

spec:
  type: NodePort         # Change to NodePort or, LoadBalancer if you need external access
  selector:
    app: nginx-web       # Must exactly match the Pod's label

  ports:
  - protocol: TCP
    port: 80             # The port the service exposes inside the cluster
    targetPort: 80       # The port the container is listening on
    nodePort: 30080      # accessible at <NodeIP>:30080
```


```
kubectl apply -f nginx-pod-svc.yaml
```


#### Example-2:

_Create a Deployment Service YAML File:_
```yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx-deployment-svc

spec:
  type: NodePort       # Defines the service type ClusterIP or LoadBalancer
  selector:
    app: web-server    # Must match the template labels in the Deployment

  ports:
  - protocol: TCP
    port: 80           # Port the Service exposes
    targetPort: 80     # Port the containers are listening on
    nodePort: 30081    # accessible at <NodeIP>:30081
```



```
kubectl apply -f nginx-deployment-svc.yaml
```

## LoadBalancer:

Creates an external Load Balancer. By changing the `type` to `LoadBalancer`, Kubernetes exposes the port externally. Traffic hitting the public IP on port `80` will be balanced across your 03 backend pods.


_Traffic Flow:_
```
Internet
    |
    v
Cloud Load Balancer / Bare Metal
    |
    v
Kubernetes Service (LoadBalancer)
    |
    v
Pods
```


**Cloud Providers:**
- Amazon Web Services (AWS)
- Google Cloud
- Microsoft Azure


**Bare Metal:**
For on-premises Kubernetes:
- MetalLB
- kube-vip


### Architecture:

```
                     Users
                       |
                       v
             192.168.10.xxx
            MetalLB VIP
                       |
          +------------+------------+
          |                         |
          v                         v
      Worker-1                 Worker-2
          |                         |
          +------------+------------+
                       |
                  Nginx Pods
```




### Imperative Method:

_Create a LoadBalancer Service directly:_
```
kubectl expose pod nginx-pod --type=LoadBalancer --port=80 --target-port=80 --name=nginx-pod-svc

kubectl expose deployment nginx-deployment --type=LoadBalancer --port=80 --target-port=80 --name=nginx-deployment-svc
```



_Generate YAML from Imperative Command:_
```
kubectl expose pod nginx-pod --type=LoadBalancer --port=80 --target-port=80 --name=nginx-pod-svc --dry-run=client -o yaml

kubectl expose deployment nginx-deployment --type=LoadBalancer --port=80 --target-port=80 --name=nginx-deployment-svc --dry-run=client -o yaml > nginx-deployment-svc.yaml
```


```
kubectl apply -f nginx-deployment-svc.yaml
```

## ExternalName:

An ExternalName Service maps a Kubernetes Service name to an external DNS name. E.g., (`my-database.external-provider.com`) by returning a `CNAME` record. It does not use selectors or target internal Pods.

- Internal Pods can call this Service, and Kubernetes will redirect them to the external destination.
- Allowing your internal apps to call an external database or third-party API using a standard Kubernetes internal URL.


```yaml
apiVersion: v1
kind: Service
metadata:
  name: postgres-db-svc
  namespace: default

spec:
  type: ExternalName
  externalName: mydb.example.com   # Must be a DNS name, not an IP address.
```


```
kubectl apply -f externalname.yaml
```


```
kubectl get svc

NAME                      TYPE           CLUSTER-IP      EXTERNAL-IP        PORT(S)        AGE
postgres-db-svc           ExternalName   <none>          mydb.example.com   <none>         3s
```


#### Create Application Deployment:


_Example application deployment:_
```yaml 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app

spec:
  replicas: 2

  selector:
    matchLabels:
      app: web-app

  template:
    metadata:
      labels:
        app: web-app

    spec:
      containers:
      - name: web-app
        image: nginx:1.25

        env:
        - name: DB_HOST
          value: postgres-db-svc

        - name: DB_PORT
          value: "5432"
```





Note: `DB_HOST=postgres-db-svc` not `DB_HOST=mydb.example.com` and Kubernetes DNS resolves: 

```
postgres-db-svc.default.svc.cluster.local
        |
        v
CNAME -> mydb.example.com
        |
        v
192.168.10.50
```

### Useful Commands:

```
-- List Services in current namespace
kubectl get services
kubectl get svc          # shorthand


-- Check pod label: 
kubectl get pods --show-labels


-- Get full details of a specific Service
kubectl describe svc <service_name>


-- See which Pod IPs are currently backing the Service
kubectl get endpoints <service_name>
kubectl get endpointslices


-- Get raw YAML/JSON of a live Service
kubectl get svc <service_name> -o yaml
kubectl get svc <service_name> -o json


-- Edit a Service live (opens in default editor)
kubectl edit svc <service_name>


-- Check kube-proxy Setting and Enable strictARP (strictARP: true):
kubectl -n kube-system get configmap kube-proxy -o yaml | egrep strictARP
kubectl -n kube-system edit configmap kube-proxy


-- Check MetalLB: 
kubectl get ipaddresspool -n metallb-system
kubectl get l2advertisement -n metallb-system


-- Delete a Service
kubectl delete svc <service_name>
```
