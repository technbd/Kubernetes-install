## Kubernetes PV and PVC: 

In Kubernetes, **PV (PersistentVolume)** and **PVC (PersistentVolumeClaim)** are used to provide **persistent storage** for pods—storage that survives pod restarts, rescheduling, or failures.


**Persistent volume (PV)** is a piece of storage provided by an administrator in a Kubernetes cluster. When a developer needs **persistent storage** for an application in the cluster, they request that storage by creating a **Persistent Volume Claim (PVC)** and then mounting the volume to a path in the pod.


### Docker Volume types:
- Volume : Managed by Docker, stored in `/var/lib/docker/volumes/` and named volume `docker vloume create my_volume`
- Bind Mounts (Host-based storage) : Bind mounts map a specific host file/folder into the container.
- tmpfs : Stored in host memory only, never written to disk



### K8s Volume types:

Kubernetes supports many volume types. They are grouped into **Ephemeral** (temporary) and **Persistent** (durable) storage.

1. Ephemeral volumes (tied to Pod lifecycle): 
    - EmptyDir (Temporary Storage): Empty directory created when Pod starts, deleted when Pod is removed. 
    - ConfigMap Volume (Used to inject configuration files)
    - Secret Volume (Used for sensitive data (passwords, tokens))

2. Persistent / network volumes:
    - HostPath (Node-level storage): Mounts a file/directory from the node's filesystem
    - Persistent Volume (PV + PVC): References a PVC — the standard way to get persistent storage
    - NFS Volume (Shared network storage)
    - Cloud Provider Volumes 
    - CSI (Container Storage Interface): Ceph CSI, Longhorn CSI, OpenEBS CSI
    - local : A mounted local disk/partition on a specific node, used via PV with node affinity



### Volumes Summary: 

| Volume Type   | Purpose                | Persistence |
| ------------- | ---------------------- | ----------- |
| emptyDir      | Temporary storage      | ❌ No        |
| hostPath      | Node filesystem access | ⚠️ Limited  |
| PV/PVC        | Persistent storage     | ✅ Yes       |
| configMap     | Config files           | ❌ No        |
| secret        | Sensitive data         | ❌ No        |
| NFS           | Shared storage         | ✅ Yes       |
| Cloud volumes | Cloud disks            | ✅ Yes       |
| CSI           | Modern storage plugins | ✅ Yes       |
| downwardAPI   | Pod metadata           | ❌ No        |

## EmptyDir:

An emptyDir volume is:
- Created when a Pod starts
- Shared between containers in the same Pod
- Deleted when the Pod is removed


```yaml 
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod-emptydir 
spec:
  containers:

  # NGINX container (serves content)
  - name: nginx-container
    image: nginx:1.25
    ports:
    - containerPort: 80
    volumeMounts:
    - name: shared-data
      mountPath: /usr/share/nginx/html

  # Writer container (creates content)
  - name: writer-container
    image: busybox
    command: ["/bin/sh", "-c"]
    args:
      - while true; do
          echo "Hello from EmptyDir... $(date)" > /data/index.html;
          sleep 5;
        done
    volumeMounts:
    - name: shared-data
      mountPath: /data

  volumes:
  - name: shared-data
    emptyDir: {}
```


```
kubectl apply -f nginx-pod-emptydir.yaml
```


```
kubectl port-forward --address 0.0.0.0 pod/nginx-pod-emptydir 80:80
```



_Open browser:_
```
http://node-ip:80
```

## HostPath: 

Mount a node directory.

_When to use HostPath:_
- Logging agents (Fluentd, Filebeat)
- Monitoring tools (node-exporter)
- Dev/test clusters
- Not for production app data


```yaml 
apiVersion: v1
kind: Pod
metadata:
  labels:
    run: nginx-pod-hostpath
  name: nginx-pod-hostpath

spec:
  containers:
  - image: nginx:1.25
    name: nginx-container
    ports:
      - containerPort: 80
    volumeMounts:
      - name: nginx-hostpath-vol
        mountPath: /usr/share/nginx/html   ## Path inside the container

  volumes:
    - name: nginx-hostpath-vol
      hostPath:
        path: /data/nginx/html   ## Directory on Kubernetes node
        type: DirectoryOrCreate  ## Creates folder if it doesn’t exist (with 0755 permissions) 
```


```
kubectl apply -f nginx-pod-hostPath.yaml
```


```
kubectl expose pod nginx-pod-hostpath --name=nginx-pod-hostpath-svc --port=80 --target-port=80 --type=NodePort
```


```
echo "Ngimx HostPath..." > /data/nginx/html/index.html
```


_Access Pod:_
```
http://node-ip:NodePort 
```

## PersistentVolume (PV) and PersistentVolumeClaim (PVC):


- PV (PersistentVolume) → “Actual storage in the cluster”
- PVC (PersistentVolumeClaim) → “Request for storage from a user/app”


### Architecture:

```
PV (Storage)  --->  PVC (Request)  --->  Pod (NGINX)
```



### PersistentVolume (PV): cluster-level resource, manually defined

A PV is a cluster-level storage resource created by an admin.

_It can come from:_
- Local disk
- NFS
- Cloud storage (EBS, Azure Disk, GCE PD, etc.)



```yaml 
apiVersion: v1
kind: PersistentVolume
metadata:
  name: nginx-pv

spec:
  capacity:
    storage: 20Gi
  accessModes:
    - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain
  storageClassName: manual
  hostPath:
    path: /data/nginx/html
    type: DirectoryOrCreate
  
  #nfs: 
    #path: /tmp   #NFS_Share
    #server: <nfs_server_ip>
```




### PersistentVolumeClaim (PVC): namespaced request for storage

A PVC is created by a user/application to request storage.

```yaml 
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: nginx-pvc
  namespace: default

spec:
  accessModes:
    - ReadWriteOnce           ## Means the volume can be mounted read-write by only one node at a time
  storageClassName: manual
  resources:
    requests:
      storage: 2Gi
```



```
kubectl apply -f pv.yaml

kubectl apply -f pvc.yaml
```


_Verify:_
```
kubectl get pv         # STATUS should show "Bound"
```

### NGINX Deployment using PVC: 


Now, deploy Nginx and tell it to use the PVC. We will mount the volume to Nginx's default HTML directory (`/usr/share/nginx/html`). Any files placed in `/data/nginx/html` on your host will be served by Nginx.


```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
  namespace: default
  labels:
    app: web-server

spec:
  replicas: 1          # ReadWriteOnce: volume can be mounted read-write by only one node at a time. Multiple replicas and they land on different nodes, extra Pods will be stuck Pending waiting for the volume
  selector:
    matchLabels:
      app: web-server  # Tells the Deployment which Pods belong to it

  template:
    metadata:
      labels:
        app: web-server # This label must match the selector above

    spec:
      containers:
      - name: nginx-container
        image: nginx:1.25
        ports:
        - containerPort: 80 

        volumeMounts:
        - name: nginx-storage
          mountPath: /usr/share/nginx/html

      volumes:
      - name: nginx-storage
        persistentVolumeClaim:
          claimName: nginx-pvc  ## pvc-name
```



```
kubectl apply -f nginx-deployment.yaml
```

```
kubectl get pod

```
