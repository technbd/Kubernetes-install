## Kubernetes PV and PVC: 

In Kubernetes, **PV (PersistentVolume)** and **PVC (PersistentVolumeClaim)** are used to provide **persistent storage** for pods‚Äîstorage that survives pod restarts, rescheduling, or failures.


**Persistent volume (PV)** is a piece of storage provided by an administrator in a Kubernetes cluster. When a developer needs **persistent storage** for an application in the cluster, they request that storage by creating a **Persistent Volume Claim (PVC)** and then mounting the volume to a path in the pod.


### Docker Volume types:
- Volume : Managed by Docker, stored in `/var/lib/docker/volumes/` and named volume `docker vloume create my_volume`
- Bind Mounts (Host-based storage) : Bind mounts map a specific host file/folder into the container.
- tmpfs : Stored in host memory only, never written to disk



### K8s Volume types:

Kubernetes supports many volume types. They are grouped into **Ephemeral** (temporary) and **Persistent** (durable) storage.

1. Ephemeral volumes (tied to Pod lifecycle): 
    - EmptyDir (Temporary Storage): Empty directory created when Pod starts, deleted when Pod is removed. 
    - ConfigMap Volume (Used to inject configuration files)
    - Secret Volume (Used for sensitive data (passwords, tokens))

2. Persistent / network volumes:
    - HostPath (Node-level storage): Mounts a file/directory from the node's filesystem
    - Persistent Volume (PV + PVC): References a PVC ‚Äî the standard way to get persistent storage
    - NFS Volume (Shared network storage)
    - Cloud Provider Volumes 
    - CSI (Container Storage Interface): Ceph CSI, Longhorn CSI, OpenEBS CSI
    - local : A mounted local disk/partition on a specific node, used via PV with node affinity

### PersistentVolume Access Modes: 

PersistentVolume Access Modes define how a volume can be mounted by nodes and pods. The Four Core Access Modes: 

| Access Mode   | Short Name | Description                 |
| ------------- | ---------- | --------------------------- |
| ReadWriteOnce | RWO        | Read-write by a single node at a time |
| ReadOnlyMany  | ROX        | Read-only by many nodes simultaneously |
| ReadWriteMany | RWX        | Read-write by many nodes simultaneously |
| ReadWriteOncePod | RWOP    | Read-write by a single Pod (not just node) cluster-wide  |



**Which storage backends support which modes:**

Block storage (EBS, GCE PD, Azure Disk, local disks) is fundamentally single-node ‚Äî they can't be mounted by multiple nodes concurrently. File-based/network storage (NFS, EFS, CephFS) supports concurrent multi-node access because the protocol itself handles concurrent access.


| Storage Backend               | RWO | ROX | RWX | RWOP |
| ----------------------------- | ----| --- | --- | ---- |
| AWS EBS / GCE PD / Azure Disk | ‚úÖ  | ‚ùå  | ‚ùå | ‚úÖ  |
| Local Storage / hostPath      | ‚úÖ  | ‚ùå  | ‚ùå | ‚úÖ  |
| NFS                           | ‚úÖ  | ‚úÖ  | ‚úÖ | ‚úÖ  |
| CephFS                        | ‚úÖ  | ‚úÖ  | ‚úÖ | ‚úÖ  |
| GlusterFS                     | ‚úÖ  | ‚úÖ  | ‚úÖ | ‚úÖ  |
| Azure Files                   | ‚úÖ  | ‚úÖ  | ‚úÖ | ‚úÖ  |
| AWS EFS                       | ‚úÖ  | ‚úÖ  | ‚úÖ | ‚úÖ  |
| iSCSI                         | ‚úÖ  | ‚úÖ  | ‚ùå | ‚úÖ  |



#### ReadWriteOnce (RWO):
A volume can be mounted as read-write by one node at a time.

Multiple pods can access the volume only if they are all running on the same node. If a pod on a different node tries to attach to it, the mount will fail. 

**Common use case**: Single-instance databases (like MySQL or PostgreSQL) where data consistency requires a single writer on a single host.

_Common Storage:_
- AWS EBS
- Azure Disk
- GCE Persistent Disk
- Longhorn (typically)

```
Node1
 ‚îú‚îÄ Pod-A
 ‚îú‚îÄ Pod-B
 ‚îî‚îÄ Pod-C

‚úì All can use the volume
```

But,
```
Node1  <-- volume mounted
Node2  <-- cannot mount same volume RW
```


#### ReadOnlyMany (ROX): 
Multiple nodes can mount the volume, but only read it.

Multiple pods across different nodes can read from the volume at the same time, but none of them can modify the data.

_Use Cases:_
- Shared documentation
- Static website content
- Software repositories

```
Node1 ‚îÄ‚îê
Node2 ‚îÄ‚îº‚îÄ‚îÄ Read Only
Node3 ‚îÄ‚îò
```


#### ReadWriteMany (RWX): 
Multiple nodes and multiple pods can read and write simultaneously.

Pods running on entirely different nodes can read from and write to the same volume at the same time. This requires a network file system or a distributed storage backend.

_Common Storage:_
- NFS
- CephFS
- GlusterFS
- Azure Files

```
Node1 ‚îÄ‚îê
Node2 ‚îÄ‚îº‚îÄ‚îÄ Read + Write
Node3 ‚îÄ‚îò
```


#### ReadWriteOncePod (RWOP):

Only one Pod in the entire cluster can mount the volume as read-write.

Introduced as a stable feature in Kubernetes 1.29, this mode ensures that only one pod in the entire cluster can read from or write to the PV.


_Useful for:_
- Databases
- Leader-election workloads
- Single-writer applications


```
Pod-A  ‚úì
Pod-B  ‚úó
Pod-C  ‚úó
```


#### Access Mode Summary:

| Access Mode | One Pod        | Multiple Pods Same Node | Multiple Nodes |
| ----------- | -------------- | ----------------------- | -------------- |
| RWO         | ‚úÖ              | ‚úÖ                       | ‚ùå              |
| ROX         | ‚úÖ Read Only    | ‚úÖ Read Only             | ‚úÖ Read Only    |
| RWX         | ‚úÖ              | ‚úÖ                       | ‚úÖ              |
| RWOP        | ‚úÖ Only One Pod | ‚ùå                       | ‚ùå              |




### Reclaim Policies:

It is a PersistentVolume Reclaim Policy. These define what happens to the PV after the PVC is deleted. 

| Policy | What happens to the PV when its PVC is deleted |
| ------ | ---------------------------------------------- |
| `Delete` | Delete the PV and its underlying storage are automatically deleted. Data is gone. | 
| `Retain` | PV is kept and data after PVC deletion. Moves to `Released` status. Data stays intact. Requires manual cleanup before reuse. | 
| `Recycle` | Deprecated. Removed in newer Kubernetes versions. | 



#### Common Production Choices:

| Storage Type            | Recommended Reclaim Policy |
| ----------------------- | -------------------------- |
| Database Storage        | Retain                     |
| Application Data        | Retain                     |
| Backups                 | Retain                     |
| CI/CD Temporary Storage | Delete                     |
| Dev/test Environment    | Delete                     |
| Shared NFS/file storage where you want a safety net | `Retain`, or `archiveOnDelete: true` if using a provisioner that supports archiving instead of straight deletion|

## EmptyDir:

An emptyDir volume is:
- Created when a Pod starts
- Shared between containers in the same Pod
- Deleted when the Pod is removed


```yaml 
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod-emptydir 
spec:
  containers:

  # NGINX container (serves content)
  - name: nginx-container
    image: nginx:1.25
    ports:
    - containerPort: 80
    volumeMounts:
    - name: shared-data
      mountPath: /usr/share/nginx/html

  # Writer container (creates content)
  - name: writer-container
    image: busybox
    command: ["/bin/sh", "-c"]
    args:
      - while true; do
          echo "Hello from EmptyDir... $(date)" > /data/index.html;
          sleep 5;
        done
    volumeMounts:
    - name: shared-data
      mountPath: /data

  volumes:
  - name: shared-data
    emptyDir: {}
```


```
kubectl apply -f nginx-pod-emptydir.yaml
```


```
kubectl port-forward --address 0.0.0.0 pod/nginx-pod-emptydir 80:80
```



_Open browser:_
```
http://node-ip:80
```

## HostPath: 

Mount a node directory.

_When to use HostPath:_
- Logging agents (Fluentd, Filebeat)
- Monitoring tools (node-exporter)
- Dev/test clusters
- Not for production app data


```yaml 
apiVersion: v1
kind: Pod
metadata:
  labels:
    run: nginx-pod-hostpath
  name: nginx-pod-hostpath

spec:
  containers:
  - image: nginx:1.25
    name: nginx-container
    ports:
      - containerPort: 80
    volumeMounts:
      - name: nginx-hostpath-vol
        mountPath: /usr/share/nginx/html   ## Path inside the container

  volumes:
    - name: nginx-hostpath-vol
      hostPath:
        path: /data/nginx/html   ## Directory on Kubernetes node
        type: DirectoryOrCreate  ## Creates folder if it doesn‚Äôt exist (with 0755 permissions) 
```


```
kubectl apply -f nginx-pod-hostPath.yaml
```


```
kubectl expose pod nginx-pod-hostpath --name=nginx-pod-hostpath-svc --port=80 --target-port=80 --type=NodePort
```


```
echo "Ngimx HostPath..." > /data/nginx/html/index.html
```

```
http://node-ip:NodePort 
```

## PersistentVolume (PV) and PersistentVolumeClaim (PVC):


- PV (PersistentVolume) ‚Üí ‚ÄúActual storage in the cluster‚Äù
- PVC (PersistentVolumeClaim) ‚Üí ‚ÄúRequest for storage from a user/app‚Äù


### Architecture:

```
PV (Storage)  --->  PVC (Request)  --->  Pod (NGINX)
```



### PersistentVolume (PV): cluster-level resource, manually defined

A PV is a cluster-level storage resource created by an admin.

_It can come from:_
- Local disk
- NFS
- Cloud storage (EBS, Azure Disk, GCE PD, etc.)



```yaml 
apiVersion: v1
kind: PersistentVolume
metadata:
  name: nginx-pv

spec:
  capacity:
    storage: 20Gi
  accessModes:
    - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain
  storageClassName: manual
  hostPath:
    path: /data/nginx/html
    type: DirectoryOrCreate
  
  #nfs: 
    #path: /tmp   #NFS_Share
    #server: <nfs_server_ip>
```




### PersistentVolumeClaim (PVC): namespaced request for storage

A PVC is created by a user/application to request storage.

```yaml 
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: nginx-pvc
  namespace: default

spec:
  accessModes:
    - ReadWriteOnce           ## Means the volume can be mounted read-write by only one node at a time
  storageClassName: manual
  resources:
    requests:
      storage: 2Gi
```



```
kubectl apply -f pv.yaml

kubectl apply -f pvc.yaml
```

### NGINX Deployment using PVC: 


Now, deploy Nginx and tell it to use the PVC. We will mount the volume to Nginx's default HTML directory (`/usr/share/nginx/html`). Any files placed in `/data/nginx/html` on your host will be served by Nginx.


```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
  namespace: default
  labels:
    app: web-server

spec:
  replicas: 1          # ReadWriteOnce: volume can be mounted read-write by only one node at a time. Multiple replicas and they land on different nodes, extra Pods will be stuck Pending waiting for the volume
  selector:
    matchLabels:
      app: web-server  # Tells the Deployment which Pods belong to it

  template:
    metadata:
      labels:
        app: web-server # This label must match the selector above

    spec:
      containers:
      - name: nginx-container
        image: nginx:1.25
        ports:
        - containerPort: 80 

        volumeMounts:
        - name: nginx-storage
          mountPath: /usr/share/nginx/html

      volumes:
      - name: nginx-storage
        persistentVolumeClaim:
          claimName: nginx-pvc  ## pvc-name
```



```
kubectl apply -f nginx-deployment.yaml
```

```
kubectl get pod

nginx-deployment-7c7dbcf46-brh5k   1/1     Running   0          8s
```
