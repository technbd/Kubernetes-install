## Kubernetes `command` and `args`:

In Kubernetes, the `command` and `args` fields in a Pod configuration file allow you to define or override the executable and parameters that run inside a container.



### CMD vs ENTRYPOINT in Dockerfile:

- The `CMD` instruction is used to define the **default command to execute when running the container**. This sets the command executed upon container creation. **Docker only allows one CMD instruction per file**. When you have more than one defined, it runs the last command.
    - **Purpose**: To specify default arguments for a container.
    - **Overridable**: Yes, the CMD can be overridden by the command passed at runtime with `docker run <image> <command>`.
    - Shell form: `CMD echo "Hello, World"`
    - Exec form : `CMD ["echo", "Hello, World"]` this exec form is parsed as a `JSON` array, which means that you **must use double-quotes (")** around words, not single-quotes (').

- The `ENTRYPOINT` instruction is used to define the first and default command that will be executed when the container is creation. Define the command to start your application with the ENTRYPOINT instruction. You can append arguments to this fixed command by providing additional arguments at runtime, but the command itself cannot be replaced unless you use the `--entrypoint` flag.
    - **Purpose**: To enforce a specific command that always runs when the container starts.
    - **Overridable**: No, the command cannot be overridden without using the `--entrypoint` flag; only the arguments can be overridden.
    - Shell form: `ENTRYPOINT echo "Hello, World"`
    - Exec form : `ENTRYPOINT ["echo", "Hello"]`



### `command` vs `args` in Kubernetes:

_They are equivalent to Docker's, though Kubernetes uses slightly different naming:_

| Kubernetes Field | Docker Equivalent | Purpose   |
| ---------------- | ----------------- | ----------|
| `command`        | `ENTRYPOINT`      | Specifies the actual executable or binary to run. | 
| `args`           | `CMD`             | Specifies the arguments/parameters passed into that executable. | 





### Override Behavior:

- **If you don't define either**: Kubernetes runs the default `ENTRYPOINT` and `CMD` from the Docker image.
- **If you define `command` but no `args`**: Only the specified `command` is run. Any default `CMD` in the Dockerfile is completely ignored.
- **If you define `args` but no `command`**: The default `ENTRYPOINT` from the Dockerfile is executed using your custom `args`.
- **If you define both**: Both the default `ENTRYPOINT` and `CMD` are completely overridden by your settings.


### Command vs Args Summary:

| Configuration           | Result                                   |
| ----------------------- | ---------------------------------------- |
| No `command`, No `args` | Use image ENTRYPOINT + CMD               |
| Only `args`             | Use image ENTRYPOINT + new args (CMD is replaced)  |
| Only `command`          | Use new command, ignore image ENTRYPOINT |
| `command` + `args`      | Use new command + new args               |




### Syntax Formats:

#### Option A: List Format (Recommended for readability):

```
command: ["echo", "Hello World"]
```



#### Option B: Multi-line / Block Format:

```
command:
  - "echo"
  - "Hello World"
```

### Example 1: Basic execution

Override the image's `ENTRYPOINT`.

When you only set `command` (no `args`), it completely overrides the image's default `ENTRYPOINT`, and the image's `CMD` is ignored entirely.

The container **prints the message and exits** (since echo finishes immediately — fine for a demo, but in real use you'd want a long-running process, or `restartPolicy: Never` if this is meant to be a one-off job). In this example, the container starts up, runs the `echo` command with its text, and then exits.  

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: demo-pod1

spec:
  containers:
  - name: demo-container
    image: busybox
    command: ["echo", "Hello from Kubernetes!"]
  #restartPolicy: Never 
```


```
kubectl apply -f pod1.yaml
```


```
kubectl logs demo-pod1
kubectl logs demo-pod1 -c demo-container
kubectl logs demo-pod1 -c demo-container --previous
```



### Example 2: Sleep loop

This keeps the container alive, printing "Running" every 5 seconds.
- This `command` alone overrides the image's `ENTRYPOINT` and ignores its `CMD`.
- No `args` field needed here since everything is passed directly inside `command`.
- `-c` flag means: stands for "**command string**." It tells the shell (`sh`, `bash`, etc.) to read the very next string argument as a **containing commands and execute it directly**, rather than trying to look for a script file on the disk. 


```yaml
apiVersion: v1
kind: Pod
metadata:
  name: demo-pod2

spec:
  containers:
  - name: demo-container
    image: busybox
    command: ["sh", "-c", "while true; do echo 'Running'; sleep 5; done"] 
```


```
kubectl apply -f pod2.yaml
```

```
kubectl exec -it demo-pod2 -- ps
```


```
kubectl logs demo-pod2 
kubectl logs demo-pod2 -c demo-container 
kubectl logs demo-pod2 -c demo-container --previous
```

### Example 5: `args` Only

Override the image's default `CMD`.

When you only set `args` (no `command`), the image's original `ENTRYPOINT` still runs, and your `args` replace the image's default `CMD`.

- `args` alone does not touch the image's `ENTRYPOINT` — it only replaces the default `CMD`.
- Useful when an image's entrypoint is already correct and you just want to change flags/arguments.
- If the image has no real `ENTRYPOINT` defined, `args`-only may not behave as expected — in that case, use `command` instead (or both).


```yaml
apiVersion: v1
kind: Pod
metadata:
  name: demo-pod5
spec:
  containers:
  - name: demo-container
    image: busybox
    args: ["echo", "Hello from args only!"]
```


Or,
```yaml 
apiVersion: v1
kind: Pod
metadata:
  name: demo-pod5
spec:
  containers:
  - name: demo-container
    image: busybox
    #args: ["sleep", "3600"]   ## Or,
    args:
      - "sleep"
      - "3600"
```


```
kubectl apply -f pod5.yaml
```


```
kubectl logs demo-pod5 
kubectl logs demo-pod5 -c demo-container 
kubectl logs demo-pod5 -c demo-container --previous
```

### Example 8: Passing arguments to a Node image (Block Format)

- The container effectively runs: `ENTRYPOINT` (`node`) + `args` (`server.js --port=3000 --environment=production`) concatenate together. 
- The root endpoint returned: `{"message":"Hello from Kubernetes!","environment":"production","port":3000,"hostname":"demo-pod8"}`


```yaml 
apiVersion: v1
kind: Pod
metadata:
  name: demo-pod8
  labels:
    app: node-app
spec:
  containers:
  - name: node-app
    image: my-node-app:latest       # Where ENTRYPOINT is ["node"]
    imagePullPolicy: IfNotPresent
    args:
      - "server.js"
      - "--port=3000"
      - "--environment=production"
    ports:
    - containerPort: 3000
```


```
kubectl apply -f pod8.yaml
```


```
kubectl logs demo-pod8 
kubectl logs demo-pod8 -c node-app
kubectl logs demo-pod8 -c node-app --previous
```


```
kubectl exec -it demo-pod8 -- curl -I localhost:3000
```


```
kubectl port-forward --address=0.0.0.0 pod/demo-pod8 3000:3000

curl -I localhost:3000
```

### Example 13:

_Verify inside the image:_
```sh
kubectl run java-test --image=eclipse-temurin:21-jre-alpine --restart=Never -it --rm -- sh 
```


```
ls -l /app
```


_Verify Java Version:_
```yaml 
apiVersion: v1
kind: Pod
metadata:
  name: demo-pod13

spec:
  containers:
  - name: java
    image: eclipse-temurin:17-jre
    command: ["java"]
    args: ["-version"]
```


**Remember**: `/app/server.jar` must exist inside the container (either baked into the image or mounted via a volume), otherwise you'll get: `Error: Unable to access jarfile /app/server.jar`. 


_Example Dockerfile:_
```Dockerfile
FROM openjdk:17-slim

WORKDIR /app

COPY server.jar /app/server.jar

CMD ["java","-jar","/app/server.jar"]
```


_Pod YAML:_
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: demo-pod13
spec:
  containers:
  - name: api-server
    image: eclipse-temurin:21-jre-alpine
    command: ["java"]
    args:
      - "-Xms256m"
      - "-Xmx512m"
      - "-jar"
      - "/app/server.jar"
      - "--server.port=8080"
```


_Or, Pod YAML:_
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: demo-pod13

spec:
  containers:
  - name: api-server
    image: openjdk:22-ea-21-slim  # eclipse-temurin:21-jre-alpine

    command: ["java"]
    args:
      - "-Xms256m"
      - "-Xmx512m"
      - "-jar"
      - "/app/server.jar"
      - "--server.port=8080"

    volumeMounts:
    - name: app-jar
      mountPath: /app/server.jar

  volumes:
  - name: app-jar
    hostPath:
      path: /opt/apps/server.jar  ## Mount the file directly 
      type: File
```


```
kubectl apply -f pod13.yaml
```


```
kubectl logs demo-pod13 
```
