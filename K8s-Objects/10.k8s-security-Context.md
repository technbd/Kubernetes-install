## Kubernetes securityContext:

In Kubernetes, a `securityContext` defines the privilege and access control settings for a Pod or Container. It controls how processes run inside the container, what Linux capabilities they have, filesystem permissions, privilege escalation, and more.


It can be set at two levels:
- **Pod-level** (`spec.securityContext`): Applies to all containers in the pod. 
- **Container-level** (`spec.containers[].securityContext`): Overrides pod-level settings for a specific container.


_Where securityContext Can Be Defined:_
| Level                     | Applies To                | Example Settings                                 |
| ------------------------- | ------------------------- | ------------------------------------------------ |
| Pod SecurityContext       | All containers in the Pod | `runAsUser`, `fsGroup`, `runAsGroup`                   |
| Container SecurityContext | Individual container      | privileged, capabilities, `readOnlyRootFilesystem` |






### Key Fields & What They Do:

Here are the most common and powerful fields you'll use to lock down your workloads:

#### 1. Running as Non-Root (`runAsNonRoot`, `runAsUser`, `runAsGroup`)
By default, containers often run as the `root` user, which is a massive security risk if a container is compromised.
- `runAsNonRoot: true`: Validates that the container image does not run as root. If it tries to, Kubernetes will refuse to start it.
- `runAsUser` / `runAsGroup`: Forcefully runs the container processes with a specific UID (User ID) and GID (Group ID), regardless of what the Dockerfile says.


#### 2. Read-Only Root Filesystem (`readOnlyRootFilesystem`)
- Setting `readOnlyRootFilesystem: true` makes the container's root file system immutable. If an attacker gains access, they cannot download malicious scripts, overwrite binaries, or modify configuration files.
- Tip: If your app needs to write temporary logs or caches, you can pair this with an `emptyDir` volume mounted at that specific path.


#### 3. Privilege Escalation (`allowPrivilegeEscalation`)
- Setting `allowPrivilegeEscalation: false` prevents a process from gaining more privileges than its parent process (e.g., prevents the use of `su` or `sudo` mechanisms). This should almost always be set to `false` in production.


#### 4. Linux Capabilities (`capabilities`)
Linux breaks down traditional root privileges into distinct units called "capabilities" (e.g., `CAP_NET_ADMIN`, `CAP_SYS_ADMIN`).

- You can drop all unnecessary privileges and only add back the exact ones your app needs. The best practice is to **drop** `ALL` and selectively add if necessary.


#### 5. Volume Ownership (`fsGroup`)
- `fsGroup` is a Pod-level setting. When Kubernetes mounts a volume, it automatically changes the ownership of that volume to match the GID specified in `fsGroup`, allowing your non-root container to read and write to it seamlessly.



### Pod-Level vs Container-Level:

| Field | Level | Purpose |
| ----- | ----- | ------- |
| `runAsUser` | Pod / Container | UID to run the process as |
| `runAsGroup`  | Pod / Container | GID for the process | 
| `runAsNonRoot` | Pod / Container | Reject if UID is 0 |
| `fsGroup` | Pod only | GID applied to mounted volumes |
| `fsGroupChangePolicy` | Pod only | `Always` or `OnRootMismatch` |
| `allowPrivilegeEscalation` | Container | Prevent sudo/setuid escalation |
| `readOnlyRootFilesystem` | Container | Immutable container filesystem |
| `privileged` | Container | Gives root-equivalent access (avoid!) |
| `capabilities.drop/add` | Container | Fine-grained Linux capabilities |
| `seccompProfile` | Pod / Container | Syscall filtering profile |
| `appArmorProfile` | Pod / Container | AppArmor profile (k8s 1.30+) |
| `seLinuxOptions` | Pod / Container | SELinux labels |
| `sysctls` | Pod only | Kernel parameter overrides |




### Example:

Here is a practical example of a Pod manifest utilizing a hardened securityContext at both the Pod and Container levels:


```yaml
apiVersion: v1
kind: Pod
metadata:
  name: secure-app-pod
spec:
  # Pod-level security context
  securityContext:
    runAsUser: 10001
    runAsGroup: 10001
    fsGroup: 20001        # Changes volume ownership to this GID
    runAsNonRoot: true
    seccompProfile:
      type: RuntimeDefault # Uses the container runtime's default secure profile
  
  containers:
  - name: web-app
    image: nginx:1.25
    
    # Container-level security context
    securityContext:
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      capabilities:
        drop:
        - ALL             # Drops all Linux root privileges
        
    volumeMounts:
    - name: cache-volume
      mountPath: /var/cache/nginx
      
  volumes:
  - name: cache-volume
    emptyDir: {}          # Writable scratch space for the read-only container
```

## Common securityContext Options:


### runAsUser: 

Runs the container as a specific Linux UID.

Without this:
```
Container
   root (UID 0)
```


With:
```
securityContext:
  runAsUser: 1001
```


Now:
```
UID = 1001
```


Verify:
```
kubectl exec -it <pod_name> -- id
```



### runAsGroup:

Sets the primary Linux group.

```
securityContext:
  runAsUser: 1001
  runAsGroup: 3001
```


Verify:
```
kubectl exec -it client -- id
```


### runAsNonRoot:

Forces Kubernetes to reject containers that run as root.

```
securityContext:
  runAsNonRoot: true
```


### fsGroup:

Changes ownership of mounted volumes.

```
securityContext:
  fsGroup: 2000
```


Mounted volume becomes:
```
Owner: root
Group: 2000
```

Processes running as UID 1001 can write if group permissions allow.

Very common with PVCs.



### allowPrivilegeEscalation:

Controls whether a process can gain additional privileges (for example through setuid binaries).

```
securityContext:
  allowPrivilegeEscalation: false
```



### readOnlyRootFilesystem:

Makes container root filesystem read-only.

```
securityContext:
  readOnlyRootFilesystem: true
```

Cannot write:
```
/
├── bin
├── etc
├── usr
```


Example:
```
touch /test.txt

Permission denied or Read-only file system
```


Applications needing writes should use mounted volumes like:
```
/tmp
/data
/cache
```




### privileged:

Runs container like a host process. Avoid unless absolutely necessary.

```
securityContext:
  privileged: true
```


### capabilities:

Linux divides root privileges into smaller units called capabilities.

Give only:
```
NET_ADMIN
```


Example:
```
securityContext:
  capabilities:
    add:
      - NET_ADMIN
```


Remove all:
```
securityContext:
  capabilities:
    drop:
      - ALL
```


Add only what is needed:
```
capabilities:
  drop:
    - ALL
  add:
    - NET_BIND_SERVICE
```



### seccompProfile:

Restricts Linux system calls.

Example:
```
securityContext:
  seccompProfile:
    type: RuntimeDefault
```

Other options: 
```
RuntimeDefault  (Recommended)
Localhost
Unconfined
```


### appArmorProfile:

If AppArmor is enabled on the node, you can specify a profile for the container.

```
securityContext:
  appArmorProfile:
    type: RuntimeDefault
```
