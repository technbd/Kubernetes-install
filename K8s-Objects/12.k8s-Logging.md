## Kubernetes Logs:

Kubernetes logs are divided into two main categories: Application logs (generated by your containers) and System logs (generated by the internal Kubernetes components). Because containers are ephemeral, Kubernetes stores these logs temporarily on the host node, requiring a centralized logging system for long-term retention.



### Types of Logs:

1. **Application Logs**:  
    - These are produced by your deployed microservices. They track runtime metrics, database queries, application errors, and user traffic transactions.

2. **System Component Logs**:
    - **Control Plane Components**: `kube-apiserver`, `kube-scheduler`, and `kube-controller-manager` typically run as static pods and write logs to` /var/log`.
    - **Node Components**: The kubelet and container runtime run directly on the host operating system and generally log to `journald` (on Linux systems).

3. **Audit Logs**:
    - A specialized ledger managed by the API server. It records a security trail of every request made to the cluster, tracking who made the call, what they changed, and when it occurred.



### Basic Commands:


#### View logs for a single pod:

```
kubectl logs <pod-name>

kubectl logs <pod-name> --tail=50

kubectl logs <pod-name> --since=1h

kubectl logs -f <pod-name>
```



#### View logs for a specific container in a multi-container Pod:

```
kubectl logs <pod-name> -c <container-name>
```



#### View logs from a previously crashed instance of a pod:

If your pod restarted, you can view the logs of the previous instance using the `-p` or `--previous` flag.

```
kubectl logs --previous <pod-name> 
kubectl logs -p <pod-name>  
```


```
kubectl logs --previous <pod-name> -c <container-name>
```



#### Logs from all pods matching a label

```
kubectl get pod --show-labels

kubectl logs -l app=my-app --all-containers=true
kubectl logs -f -l app=my-app --all-containers=true
```



#### Logs from a deployment: 

```
kubectl logs deployment/<deployment-name>
```

## Centralized Logging:

### Node-Level Logging Agent:

A specialized logging agent (like **Fluentd**, **Fluent Bit**, **Promtail**, **Filebeat** or **OpenTelemetry Collector**) is deployed on every node via a DaemonSet. It tails the local `/var/log/containers/` directory and ships logs off-cluster



### Sidecar Container Agent:

A secondary container is bundled alongside your application container inside the same Pod. The sidecar reads custom application log files from a shared volume and formats/ships them.



### Direct Push from App:

The application code bypasses the container runtime entirely by utilizing an embedded SDK or logging (e.g., a Log4j) framework to stream events directly to an external logging endpoint.

## Popular K8s Logging Stacks


###  1. EFK Stack (most popular self-hosted):

#### Elasticsearch + Fluentd/Fluent Bit + Kibana:

- Fluent Bit runs as a DaemonSet on every node, tailing container logs
- Ships to Elasticsearch for storage and indexing
- Kibana provides the UI for search and dashboards
- Best for: teams wanting full control, on-prem or air-gapped environments.




### 2. Grafana Loki + Promtail:

#### Lightweight, label-based log aggregation:

- Promtail (DaemonSet) scrapes logs and pushes to Loki
- Grafana queries and visualizes — pairs naturally with Prometheus metrics
- Much cheaper storage than Elasticsearch (doesn't full-index content)
- Best for: teams already using Grafana/Prometheus, cost-conscious setups.


### 3. Managed / SaaS Solutions:

- Datadog :  Agent DaemonSet, rich K8s integration
- Splunk : Enterprise-grade, expensive 
- Elastic Cloud : Managed EFK 
- AWS CloudWatch : Native on EKS
- Google Cloud Logging : Native on GKE
