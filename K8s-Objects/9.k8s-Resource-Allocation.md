## Kubernetes Resource Allocation:

Kubernetes Resource Allocation is the process of assigning CPU, memory, storage, and other resources to Pods so that applications run efficiently without affecting each other.

Resource Allocation ensures application performance, cluster stability, and cost optimization. 




### Resource Types:

_The most common resources are:_

| Resource           | Description                            |
| ------------------ | -------------------------------------- |
| CPU                | Processing power (cores or millicores) |
| Memory             | RAM                                    |
| Ephemeral Storage  | Local disk used by containers          |
| Persistent Storage | PVC/PV volumes                         |
| GPU                | Hardware accelerators                  |
| HugePages          | Large memory pages                     |




### CPU Allocation:

_CPU is measured in:_
```
1 CPU = 1000m (millicores)

1m    = 0.001 CPU
10m	  = 0.01 CPU
100m	= 0.1 CPU
250m	= 0.25 CPU
500m	= 0.5 CPU
1000m	= 1 CPU
2000m	= 2 CPUs
4000m = 4 CPUs
```


_Example:_
```
resources:
  requests:
    cpu: "500m"
  limits:
    cpu: "1"
```


_Meaning:_
- Scheduler reserves 0.5 CPU
- Container can use up to 1 CPU





### Memory Allocation:

Memory uses bytes.

```
Mi = Mebibytes
Gi = Gibibytes

128Mi
512Mi
1Gi
4Gi
```


_Example:_
```
resources:
  requests:
    memory: "512Mi"
  limits:
    memory: "1Gi"
```


_Meaning:_
- Scheduler reserves 512Mi
- Container may consume 1Gi

## Core Concepts:

### 1. Requests vs Limits:

This is the most important concept.


| -        | Requests | Limits |
| -        | -------- | ------ |
| Purpose  | Minimum guaranteed resources | Maximum allowed resources | 
| Used for | Scheduling decisions | Runtime enforcement | 
| Effect if exceeded | Pod won't be scheduled | CPU throttled; container OOMKilled| 


#### Request:

Minimum guaranteed resources. Scheduler uses requests to place Pods.

_Example:_
```
requests:
  cpu: 500m
  memory: 512Mi
```


_Pod requires:_
- 0.5 CPU
- 512Mi RAM

The scheduler only places the Pod on nodes that have at least those resources available.




#### Limit:

Maximum resources a container can use.


_Example:_
```
limits:
  cpu: "1"
  memory: 1Gi
```


_Container cannot exceed:_
- 1 CPU
- 1Gi RAM




#### Example: 

```
Node capacity:
CPU: 4
Memory: 8Gi


Pod A: 
requests:
  cpu: 1
  memory: 2Gi


Remaining:
CPU: 3
Memory: 6Gi


Pod B:
requests:
  cpu: 2
  memory: 4Gi


Remaining:
CPU: 1
Memory: 2Gi



Pod C: 
requests:
  cpu: 2
  memory: 2Gi



Scheduler says: 
- Cannot schedule
- because only 1 CPU remains.
```




#### Node Allocatable:

A node's total resources are not fully available for Pods because Kubernetes reserves some for the operating system and Kubernetes components.

```
Node capacity:
8 CPU
32Gi RAM


Reserved:
OS        1 CPU
Kubelet   500m
System    2Gi



Allocatable:
CPU: 6.5
Memory: 30Gi


The scheduler uses Allocatable, not the node's raw capacity.
```


_Check it with:_
```
kubectl describe node worker1
```

### Pod with Requests & Limits:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: node-app

spec:
  containers:
    - name: node-app-ct
      image: technbd/node-app
      ports:
        - containerPort: 3000

      resources:
        requests:
          cpu: "250m"       # 0.25 vCPU guaranteed
          memory: "128Mi"
        limits:
          cpu: "500m"       # 0.5 vCPU max
          memory: "256Mi"   # Hard memory ceiling
```



```
kubectl apply -f node-app.yaml
```



```
kubectl describe pod node-app


### Output:

    Limits:
      cpu:     500m
      memory:  256Mi
    Requests:
      cpu:        250m
      memory:     128Mi
```




### Deployment with Requests & Limits:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: node-app
  labels:
    app: node-app

spec:
  replicas: 2          # Tells Kubernetes to run 3 identical copies of this Pod
  selector:
    matchLabels:
      app: node-app  # Tells the Deployment which Pods belong to it

  template:
    metadata:
      labels:
        app: node-app # This label must match the selector above

    spec:
      containers:
      - name: node-app-ct
        image: technbd/node-app
        ports:
        - containerPort: 3000

        resources:
          requests:
            cpu: "200m"       # 0.25 vCPU guaranteed
            memory: "128Mi"
          limits:
            cpu: "500m"       # 0.5 vCPU max
            memory: "256Mi"   # Hard memory ceiling
```



```
kubectl apply -f node-app.yaml --dry-run=client

kubectl apply -f node-app.yaml 
```

### 2. Quality of Service (QoS):

Kubernetes assigns each Pod a QoS class based on its requests and limits.


| QoS        | Condition                                                              | Priority During Memory Pressure |
| ---------- | ---------------------------------------------------------------------- | ------------------------------- |
| Guaranteed | Requests = Limits for every CPU and memory resource in every container | Highest                         |
| Burstable  | At least one request is set, but requests ≠ limits                     | Medium                          |
| BestEffort | No requests and no limits                                              | Lowest (First to be killed)     |



#### Guaranteed:

Most protected from eviction.

```
resources:
  requests:
    cpu: 1
    memory: 1Gi
  limits:
    cpu: 1
    memory: 1Gi
```



#### Burstable:

Can burst beyond the requested resources up to the configured limits.

```
resources:
  requests:
    cpu: 500m
    memory: 512Mi
  limits:
    cpu: 2
    memory: 2Gi
```


#### BestEffort:
No resources defined. First Pods to be evicted when the node runs out of memory.

```
containers:
  - name: node-app-ct  
    image: technbd/node-app
```








### 3. LimitRange:

Provides default and minimum/maximum resource values for Pods and containers in a namespace.


```yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: default-limits

spec:
  limits:
  - type: Container
    default:
      cpu: "500m"
      memory: "512Mi"
    defaultRequest:
      cpu: "250m"
      memory: "128Mi"
    
```


```
kubectl apply -f default-limits.yaml --dry-run=client
```



### 4. ResourceQuota: 

Limits total resource consumption within a namespace. Namespace users cannot exceed these totals.

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: dev-quota
  namespace: dev

spec:
  hard:
    requests.cpu: "4"          # Total CPU requests in namespace
    requests.memory: 8Gi
    limits.cpu: "8"
    limits.memory: 16Gi
    pods: "20"               # Max number of pods
```



```
kubectl apply -f dev-quota.yaml --dry-run=client
```



### 5. VPA & HPA:


#### HPA:

A Horizontal Pod Autoscaler (HPA) automatically increases or decreases the number of Pod replicas based on resource usage (CPU, memory) or custom metrics.


#### Prerequisites:
- HPA requires the Metrics Server.


_Verify it:_
```
kubectl get apiservices | grep metrics

v1beta1.metrics.k8s.io              kube-system/metrics-server   True        3h10m
```


Or,
```
kubectl top nodes
kubectl top pods
```



_Create a Deployment:_
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: node-app
  labels:
    app: node-app

spec:
  replicas: 2          # Tells Kubernetes to run 3 identical copies of this Pod
  selector:
    matchLabels:
      app: node-app  # Tells the Deployment which Pods belong to it

  template:
    metadata:
      labels:
        app: node-app # This label must match the selector above

    spec:
      containers:
      - name: node-app-ct
        image: technbd/node-app
        ports:
        - containerPort: 3000

        resources:
          requests:
            cpu: "10m"
            memory: "128Mi"
          limits:
            cpu: "20m"
            memory: "256Mi"
 
```


```
kubectl apply -f node-app.yaml
```


_Wait for the rollout:_
```
kubectl rollout status deployment/node-app
```


_Expose deployment:_
```
kubectl expose deployment node-app --name=node-app-svc --type=NodePort --port=3000 --target-port=3000
```



_Create HPA:_
```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: node-app-hpa

spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: node-app   ## Name of Deployment

  minReplicas: 2
  maxReplicas: 5

  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70   # Scale when avg CPU > 70% of request
```



```
kubectl apply -f node-app-hpa.yaml
```



```
kubectl get hpa

NAME           REFERENCE             TARGETS        MINPODS   MAXPODS   REPLICAS   AGE
node-app-hpa   Deployment/node-app   cpu: 24%/70%   2         5         5          115m
```


```
kubectl describe hpa node-app-hpa
```
