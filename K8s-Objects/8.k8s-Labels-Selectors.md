## Kubernetes Labels and Selectors:

Labels and selectors are fundamental Kubernetes concepts used to organize, identify, and group resources.



## Labels: 

Labels are **key-value pairs** attached to objects (Pods, Services, Deployments, Nodes, etc.) for identification purposes.


#### Why Use Labels?
- Organize resources
- Filter objects
- Select Pods for Services
- Manage Deployments and ReplicaSets


#### Rules for labels:
- Must be 63 characters or less (can be empty)
- Unless empty, must begin and end with an alphanumeric character (a-z0-9A-Z)
- Could contain dashes (-), underscores (_), dots (.), and alphanumerics between.



#### Example labels:
- release: stable, release: canary
- environment: dev, environment: qa, environment: prod 
- tier: frontend, tier: backend, tier: api
- partition: customerA, partition: customerB
- track: daily, track: weekly



### Example: Pod Definition with Labels:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
  labels:            # <-- Labels are defined here
    app: nginx
    tier: frontend
    environment: dev
spec:
  containers:
  - name: nginx-container
    image: nginx:1.25
    ports: 
    - containerPort: 80
```


```
kubectl apply -f nginx-pod.yaml
```


_Show labels in output:_
```
kubectl get pods --show-labels

NAME        READY   STATUS    RESTARTS   AGE   LABELS
nginx-pod   1/1     Running   0          31m   app=nginx,environment=dev,tier=frontend
```



_Get pods with a specific label:_
```
kubectl get pods -l app=nginx

NAME        READY   STATUS    RESTARTS   AGE
nginx-pod   1/1     Running   0          31m
```


_Multiple conditions (`AND`):_
```
kubectl get pods -l app=nginx,environment=dev

NAME        READY   STATUS    RESTARTS   AGE
nginx-pod   1/1     Running   0          31m
```



_Add/remove a label on the fly:_
```
kubectl label pods nginx-pod tier-
kubectl label pods nginx-pod tier=backend
```




### Example: Deployment Definition with Labels:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
  labels:                  # <-- Labels are defined here
    app: nginx-web
    tier: frontend
    environment: prod

spec:
  replicas: 2
  selector:
    matchLabels:
      app: nginx-web  # Tells the Deployment which Pods belong to it
      environment: prod

  template:
    metadata:
      labels:
        app: nginx-web # This label must match the selector above
        environment: prod

    spec:
      containers:
      - name: nginx-container
        image: nginx:1.25
        ports:
        - containerPort: 80
        resources: {}
```



```
kubectl apply -f nginx-deployment.yaml
```

## Label Selectors: 

A selector identifies a set of Kubernetes objects based on their labels.


_Selectors are used by:_
- Services
- ReplicaSets
- Deployments
- Jobs
- Network Policies


Selectors let you query objects by their labels. There are two types:
1. Equality-based Selectors
2. Set-based Selectors (more expressive)



### Equality-based Selectors: 

Filters objects based on exact label matches (`=`, `==`, `!=`).  
  - `environment = production` : Finds all resources where the environment is production.
  - `tier != frontend` : Finds all resources that do not have a frontend tier.


_Get pods with a specific label:_
```
kubectl get pods -l app=nginx

NAME        READY   STATUS    RESTARTS   AGE
nginx-pod   1/1     Running   0          35m
```



_Multiple conditions (`AND`):_
```
kubectl get pods -l app=nginx,environment=dev

NAME        READY   STATUS    RESTARTS   AGE
nginx-pod   1/1     Running   0          36m
```



_Example `!=`:_
```
kubectl get pods -l environment!=dev

NAME                                READY   STATUS    RESTARTS   AGE
nginx-deployment-5b77bf99fc-jbdhp   1/1     Running   0          107s
nginx-deployment-5b77bf99fc-n7pk7   1/1     Running   0          107s
```





### Set-based Selectors: 

Filters objects based on a set of values (`in`, `notin`, `exists`, `DoesNotExist`).
  - `environment in (production, staging)`
  - `tier notin (frontend)`


_Example `in`:_
```
kubectl get pods -l 'environment in (dev,prod)'

NAME                                READY   STATUS    RESTARTS   AGE
nginx-deployment-5b77bf99fc-jbdhp   1/1     Running   0          2m1s
nginx-deployment-5b77bf99fc-n7pk7   1/1     Running   0          2m1s
nginx-pod                           1/1     Running   0          54m
```


_Example `notin`:_
```
kubectl get pods -l 'environment notin (dev)'

NAME                                READY   STATUS    RESTARTS   AGE
nginx-deployment-5b77bf99fc-jbdhp   1/1     Running   0          2m14s
nginx-deployment-5b77bf99fc-n7pk7   1/1     Running   0          2m14s
```


_Example `Exists`:_
```
kubectl get pods -l 'environment'

NAME                                READY   STATUS    RESTARTS   AGE
nginx-deployment-5b77bf99fc-jbdhp   1/1     Running   0          2m25s
nginx-deployment-5b77bf99fc-n7pk7   1/1     Running   0          2m25s
nginx-pod                           1/1     Running   0          54m
```


_Example `Does Not Exist`:_
```
kubectl get pods -l '!tier'

NAME                                READY   STATUS    RESTARTS   AGE
nginx-deployment-5b77bf99fc-jbdhp   1/1     Running   0          3m23s
nginx-deployment-5b77bf99fc-n7pk7   1/1     Running   0          3m23s
```




### Example: Connecting a Service to Pods

Service that uses a `selector` to **target any Pod labeled** with `app: nginx` and `environment: dev`.

```yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx-pod-svc

spec:
  selector:          
    app: nginx          # routes traffic to any Pod with this label
    environment: dev

  ports:
    - protocol: TCP
      port: 80          # The port the service exposes inside the cluster
      targetPort: 80    # The port the container is listening on
  
  type: ClusterIP       # Change to NodePort if you need external access
```



```
kubectl apply -f nginx-pod-svc.yaml
```


```
kubectl get svc

kubernetes      ClusterIP   10.96.0.1       <none>        443/TCP   13h

```



```yaml 
  name: mysql-statefulset
  labels: 
    app: mysql    ## Pods label, It manages only Pods

spec:
  serviceName: mysql-headless-svc

  replicas: 2

  selector:
    matchLabels:
```

## Kubernetes Node Selectors, Node Affinity and Taints/Tolerations:

Kubernetes gives you several mechanisms to control which pods land on which nodes. 


### Node Selector (The Simple Match):

#### Label a node: 

```
kubectl get nodes

NAME      STATUS   ROLES           AGE   VERSION
master    Ready    control-plane   35h   v1.35.6
worker1   Ready    <none>          23h   v1.35.6
worker2   Ready    <none>          17h   v1.35.6
```



_Label a node:_
```
kubectl label nodes worker1 size=large disk=ssd environment=prod 

kubectl label nodes worker2 size=large disk=hdd environment=dev 
```



_Verify:_
```
kubectl get nodes --show-labels

kubectl describe node worker1
```



_If node label add/remove:_
```
kubectl label nodes worker2 disk-
kubectl label nodes worker2 disk=hdd
```


#### Node Selector for Pod: 

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
  labels:            # <-- Labels are defined here
    app: nginx
    tier: frontend

spec:
  containers:
  - name: nginx-container
    image: nginx:1.25
    ports:
    - containerPort: 80
  nodeSelector:
    disk: ssd          # The Pod will ONLY land on nodes with this label
    environment: prod
```



```
kubectl apply -f nginx-pod.yaml
```


```
kubectl get pod -w -o wide

nginx-pod   1/1     Running   0          10s   10.244.235.137   worker1   <none>           <none>
```

### Taints & Tolerations:

While Node Selector and Affinity attract Pods to Nodes, Taints do the opposite—they allow a Node to repel a set of Pods. A taint on a node means "don't schedule here unless you tolerate me". Tolerations on a pod are what let it ignore a specific taint.


**Taints and Tolerations are Kubernetes scheduling features used to control which Pods can be scheduled on specific nodes.**
- **Taint** : Applied to a Node to repel Pods. (পডগুলোকে বিতাড়িত করার জন্য একটি নোডে প্রয়োগ করা হয়।)
- **Toleration** : Applied to a Pod to allow it to be scheduled onto a tainted node. (একটি tainted নোডে পডটিকে শিডিউল করার অনুমতি দেওয়ার জন্য এটি প্রয়োগ করা হয়।)



_Three effects are possible:_
- `NoSchedule` : won't schedule new pods unless they tolerate it. Already running pods are unaffected.
- `PreferNoSchedule` : soft version, Soft version of NoSchedule. Scheduler tries to avoid placing pods here, but will if no other option exists. 
- `NoExecute` : Strongest effect. Blocks new pods AND evicts already running pods that don't tolerate it. Can set a `tolerationSeconds` to delay eviction.


_Two Toleration Operators:_
- `Equal` — key, value, effect must all match exactly
- `Exists` — only key needs to match, value is ignored



#### Taint a node:

_Syntax:_
```
kubectl taint node <node-name> key=value:effect
```



```
kubectl taint node worker1 dedicated=gpu:NoSchedule
```



_Verify:_
```
kubectl describe node worker1

Or,

kubectl describe node worker1 | grep Taints

Taints:             dedicated=gpu:NoSchedule
```



_Remove a Taint:_
```
kubectl taint node worker1 dedicated=gpu:NoSchedule-
```



#### Example: With Toleration using GPU workload pod 

In this example, a toleration does **only one thing** : it allows a pod to schedule on a **tainted node**. It does **not force or attract** the pod to that node. `worker1` is still just one of the valid candidates, **not a required destination**.

- `worker1` has taint `dedicated=gpu:NoSchedule` — your pod has the matching toleration, so `worker1` is now **allowed but not required**.
- `worker2` has **no taint** at all — your pod can schedule here too, with zero restrictions.


The scheduler saw **two valid nodes** and picked `worker2` — possibly because it had more available resources, lower utilization, or was simply chosen by the scoring algorithm. There was nothing in the pod spec telling the scheduler to **prefer or require** `worker1`.



```yaml 
apiVersion: v1
kind: Pod
metadata:
  name: gpu-pod

spec:
  tolerations:
  - key: dedicated
    operator: Equal
    value: gpu
    effect: NoSchedule    # Must match the taint's effect exactly

  containers:
  - name: ml-training
    image: tensorflow/tensorflow:latest-gpu
    command: ["sleep", "3600"]
#    resources:
#      limits:
#        nvidia.com/gpu: 1
```



```
kubectl apply -f gpu-pod.yaml
```


```
kubectl get pod -o wide

NAME      READY   STATUS    RESTARTS   AGE   IP              NODE      NOMINATED NODE   READINESS GATES
gpu-pod   1/1     Running   0          3s    10.244.189.85   worker2   <none>           <none>
```





#### Example: With Toleration and nodeSelector using GPU workload pod 

`worker1` → has taint `dedicated=gpu:NoSchedule`, your pod tolerates it ✅. But then `nodeSelector` or `nodeAffinity` required checks for label `dedicated=gpu` — and `worker1` doesn't have that label, only the taint. **Taint key and node label are two completely separate things**.   

Although, both use the key `dedicated`, **they are independent**.



_Add the label to worker1:_
```
kubectl label nodes worker1 dedicated=gpu
```



_Using nodeSelector:_
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: gpu-pod

spec:
  tolerations:
  - key: dedicated
    operator: Equal
    value: gpu
    effect: NoSchedule        # allows scheduling on tainted worker1

  nodeSelector:
    dedicated: gpu          # forces scheduling onto worker1

  containers:
  - name: ml-training
    image: tensorflow/tensorflow:latest-gpu
    command: ["sleep", "3600"]
```


```
kubectl apply -f gpu-pod.yaml
```


```
kubectl get pod -o wide

NAME      READY   STATUS    RESTARTS   AGE   IP               NODE      NOMINATED NODE   READINESS GATES
gpu-pod   1/1     Running   0          3s    10.244.235.163   worker1   <none>           <none>
```
