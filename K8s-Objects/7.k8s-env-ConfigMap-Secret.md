## Kubernetes Environment Variables:

Environment variables are commonly used to pass configuration values (database host, ports, credentials, application settings, etc.) into containers.

There are three primary ways to define environment variables in a Pod configuration: **hardcoded values**, pulling from **ConfigMaps**, or pulling from **Secrets**. 




| Use Case                           | Recommended Method        |
| ---------------------------------- | ------------------------- |
| Static values                      | `env`                     |
| Non-sensitive configuration        | `ConfigMap`               |
| Passwords/API keys                 | `Secret`                  |
| Many variables from ConfigMap      | `envFrom`                 |
| Pod metadata (IP, Name, Namespace) | `fieldRef` (Downward API) |

## ConfigMap Imperative Commands: 


### Create from literal values: 

Great for quick, individual environment variables.

_Syntax:_
```
kubectl create configmap <configmap_name> --from-literal=<key1>=<value1> --from-literal=<key2>=<value2>
```


```
kubectl create configmap nginx-app-config-map --from-literal=APP_ENV=production --from-literal=LOG_LEVEL=debug -o yaml --dry-run=client
```


```
kubectl describe configmaps nginx-app-config-map
```



### Create from a single file:

Kubernetes will use the filename as the key and the entire contents of the file as the value.

- Key: app.properties
- Value: Entire file contents



_Suppose a file named `app.properties`:_
```
APP_NAME=myapp
APP_ENV=production
LOG_LEVEL=debug
APP_PORT=8080
```


```
kubectl create configmap nginx-app-config-map --from-file=app.properties -o yaml --dry-run=client
```


```
kubectl describe configmaps nginx-app-config-map
```




### Create from an `env` file:

If you have a standard `.env` file, this flag automatically parses the **key-value** pairs inside it.


_The main difference is how Kubernetes stores the file contents in the ConfigMap:_
| Feature              | `--from-file=file`                                                     | `--from-env-file=file`              |
| -------------------- | ---------------------------------------------------------------------- | ----------------------------------- |
| Expected file format | Any file                                                               | `KEY=VALUE` format                  |
| ConfigMap keys       | Filename becomes the key                                               | Each `KEY` becomes a separate key   |
| ConfigMap values     | Entire file content                                                    | Individual values                   |
| Best for             | Configuration files (`nginx.conf`, `my.cnf`, `application.properties`) | Environment variable files (`.env`) |




_Suppose a file named `app.env`:_
```
APP_NAME=myapp
APP_ENV=production
LOG_LEVEL=debug
APP_PORT=8080
```


```
kubectl create configmap nginx-app-config-map --from-env-file=app.env -o yaml --dry-run=client
```



### Create a ConfigMap from a Directory: 

This grabs every file in the specified folder and packages them into a single ConfigMap (each file becomes a key).

_Directory structure:_
```
config/
├── app.properties
├── database.conf
└── nginx.conf
```


```
kubectl create configmap nginx-app-config-map --from-file=config/ -o yaml --dry-run=client
```

## ConfigMap: 

A ConfigMap is a dictionary of **key-value pairs** that store plain text  configuration settings for an applications. Kubernetes pods can use ConfigMaps as configuration files, environment variables or command-line arguments.

- First, create a ConfigMap for a cluster.
- Second, consume to ConfigMap in your Pods/Deployments and use its values.

## Secret:

Same idea as ConfigMaps, but values get base64-encoded automatically.

`Note`: Secret values must be **base64-encoded** in the YAML definition (e.g., `MySecretData` encoded is `TXlTZWNyZXREYXRh`).


```
echo -n "MySecretData" | base64

echo -n "TXlTZWNyZXREYXRh" | base64 -d
```



### Create a Secret from Literal Values:

_Syntax:_
```
kubectl create secret generic <secret_name> --from-literal=<key1>=<value1>
```


```
kubectl create secret generic mysql-secret --from-literal=MYSQL_ROOT_PASSWORD=admin123 -o yaml --dry-run=client
kubectl create secret generic mysql-secret --from-literal=MYSQL_ROOT_PASSWORD=admin123 --from-literal=DB_PORT=3306 -o yaml --dry-run=client
```


### Create a Secret from a File:


- Key: filename
- Value: file contents (base64-encoded)


_Suppose a file named `db_password.txt`:_
```
MYSQL_ROOT_PASSWORD=admin
DB_PORT=3306
```


_Create the Secret:_
```
kubectl create secret generic mysql-secret --from-file=db_password.txt -o yaml --dry-run=client
```


```
kubectl get secret
```


```
kubectl describe secret mysql-secret
```





### Create a Secret from an `env` File:

_Suppose `app.env` contains:_
```
DB_USER=admin
MYSQL_ROOT_PASSWORD=admin
DB_PORT=3306
```


_Create the Secret:_
```
kubectl create secret generic mysql-secret --from-env-file=app.env -o yaml --dry-run=client
```

### Example-6: MySQL StatefulSet

This example creates a **2-replica StatefulSet, but it does not configure MySQL replication**. Both **Pods will be independent MySQL servers** with separate storage and no replication, which can lead to **inconsistent data**.

In production, MySQL is typically deployed with a **StatefulSet and a PersistentVolumeClaim (PVC)** so that database data persists across Pod restarts and rescheduling.


### Example-7: MySQL StatefulSet with Replication 


#### ConfigMap for primary/replica `my.cnf`:


```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: mysql-configmap

data:
  primary.cnf: |
    [mysqld]
    server-id=1
    log-bin=mysql-bin
    binlog_format=ROW
    skip-name-resolve
    log_replica_updates=ON
    gtid_mode=ON
    enforce_gtid_consistency=ON
  replica.cnf: |
    [mysqld]
    server-id=2
    log-bin=mysql-bin
    binlog_format=ROW
    relay-log=relay-bin
    read-only=ON
    super_read_only=ON
    skip-name-resolve
    log_replica_updates=ON
    gtid_mode=ON
    enforce_gtid_consistency=ON
```

```
kubectl apply -f mysql-configmap.yaml
```

#### Create the StatefulSet (1 master + 1 slave):

```yaml 
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: mysql-statefulset
  labels: 
    app: mysql

spec:
  serviceName: mysql-headless-svc

  replicas: 2
  
  selector:
    matchLabels:
      app: mysql

  template:
    metadata:
      labels:
        app: mysql

    spec:
      initContainers:           # Init Container
        - name: init-mysql
          image: mysql:8.4
          command:
            - bash
            - "-c"
            - |
              ordinal=${HOSTNAME##*-}
              if [ "$ordinal" = "0" ]; then
                cp /mnt/config-map/primary.cnf /mnt/conf.d/server.cnf
              else
                cp /mnt/config-map/replica.cnf /mnt/conf.d/server.cnf
              fi
          volumeMounts:
            - name: mysql-conf
              mountPath: /mnt/conf.d

            - name: mysql-config-map
              mountPath: /mnt/config-map

      containers:                 # MySQL Container
        - name: mysql-container
          image: mysql:8.4 
          imagePullPolicy: IfNotPresent
          ports:
            - containerPort: 3306

          envFrom:
            - secretRef:
                name: mysql-secret    ## mysql-secret resource 


          volumeMounts:
            - name: mysql-data
              mountPath: /var/lib/mysql

            - name: mysql-conf
              mountPath: /etc/mysql/conf.d

      volumes:
        - name: mysql-conf
          emptyDir: {}

        - name: mysql-config-map
          configMap:
            name: mysql-configmap   ## mysql-configmap resource 

  volumeClaimTemplates:          # Persistent Storage
    - metadata:
        name: mysql-data
      spec:
        accessModes: ["ReadWriteOnce"]
        resources:
          requests:
            storage: 5Gi
 
```

### Manual Replication Setup:

#### Create the replication user on the master (`mysql-0`):


```
CREATE USER 'repl'@'%' IDENTIFIED BY 'ReplPass123';

GRANT REPLICATION SLAVE ON *.* TO 'repl'@'%';

FLUSH PRIVILEGES;
```


_Lock the master and check binlog position:_
```
FLUSH TABLES WITH READ LOCK;

SHOW BINARY LOGS;
SHOW BINARY LOG STATUS;
```



#### Configure `mysql-1`:


```
kubectl exec -it mysql-statefulset-1 -- bash

cat /etc/resolv.conf

search default.svc.cluster.local svc.cluster.local cluster.local
nameserver 10.96.0.10
options ndots:5
```


_Check DNS:_
```
getent hosts mysql-statefulset-0.mysql-headless-svc

10.244.182.43   mysql-statefulset-0.mysql-headless-svc.default.svc.cluster.local
```



```
CHANGE REPLICATION SOURCE TO
SOURCE_HOST='mysql-statefulset-0.mysql-headless-svc',
SOURCE_USER='repl',
SOURCE_PASSWORD='ReplPass123',
SOURCE_AUTO_POSITION=1,
GET_SOURCE_PUBLIC_KEY=1;
```


```
START REPLICA;
```


```
SHOW REPLICA STATUS\G
```


_You want to see:_
```
Replica_IO_Running: Yes

```
