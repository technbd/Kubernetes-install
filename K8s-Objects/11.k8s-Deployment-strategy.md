## Kubernetes Deployment Strategies:

Kubernetes deployment strategies define how a new version of an application is released while minimizing downtime and risk.


_The most common deployment strategies are:_

1. Recreate
2. Rolling Update (Default)
3. Blue-Green
4. Canary
5. A/B Testing
6. Shadow (Traffic Mirroring)
7. Rolling Back (For Plan B)




### Strategy Comparison:

| Strategy       | Downtime    | Rollback                            | Resource Cost | Complexity | Best For                                       |
| -------------- | ----------- | ----------------------------------- | ------------- | ---------- | ---------------------------------------------- |
| Recreate       | ❌ Yes       | Easy/Slow                           | Low           | Low        | Development, maintenance windows               |
| Rolling Update | ✅ No        | Easy/Slow                           | Low           | Low        | General production workloads                   |
| Blue-Green     | ✅ Near-zero | Instant                             | High          | Medium     | Critical production systems                    |
| Canary         | ✅ No        | Easy/Fast                           | Medium        | High       | Incremental production releases                |
| A/B Testing    | ✅ No        | Depends                             | Medium        | High       | Feature experiments and user segmentation      |
| Shadow         | ✅ No        | N/A (users stay on current version) | High          | High       | Pre-release validation with production traffic |



### Which Strategy Should You Use?

| Scenario                                                          | Recommended Strategy |
| ----------------------------------------------------------------- | -------------------- |
| Most web applications                                             | Rolling Update       |
| Simple internal application                                       | Recreate             |
| Zero-downtime enterprise release                                  | Blue-Green           |
| Gradual rollout with monitoring                                   | Canary               |
| Compare user behavior between features                            | A/B Testing          |
| Validate a new version using live traffic without affecting users | Shadow               |

### 1. Recreate Strategy:

The Recreate strategy terminates all existing Pods before creating new ones. Stop all existing pods, then start new ones. Causes downtime.


#### Advantages:
- Very simple
- No version mixing
- Good for database migrations requiring exclusive access


#### Disadvantages:
- Downtime occurs
- Users cannot access the application during deployment


#### Best Use Cases:
- Development environments
- Internal applications
- Stateful applications requiring exclusive access

#### Workflow:

```
Version 1 (3 Pods)

Pod-1
Pod-2
Pod-3

↓

Delete all Pods

↓

Create Version 2

Pod-1
Pod-2
Pod-3
```



#### Deployment YAML:

```
strategy:
  type: Recreate
```



#### Example: 


```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: deploy-recreate
  labels:
    type: deploy-recreate

spec:
  replicas: 5
  strategy:
    type: Recreate   # ALL old pods terminated before new ones start

  selector:
    matchLabels:
      app: deploy-recreate

  template:
    metadata:
      labels:
        app: deploy-recreate

    spec:
      containers:
      - image: kodekloud/webapp-color:v1
      #- image: kodekloud/webapp-color:v2
        name: deploy-recreate-ct
        ports:
        - containerPort: 8080
```


```
kubectl apply -f deploy-recreate.yaml
```



```
kubectl describe deployment deploy-recreate | grep Image
```


```
kubectl port-forward --address=0.0.0.0 deployments/deploy-recreate 80:8080
```



_Check rollout history:_
```
kubectl rollout history deployment/deploy-recreate
```


_If the new version has issues:_
```
kubectl rollout undo deployment/deploy-recreate 
```


_Monitor the rollback:_
```
kubectl rollout status deployment/deploy-recreate
```

### 2. Rolling Update (Default) Strategy:

This is Kubernetes' default deployment strategy. Instead of replacing everything at once, Kubernetes gradually replaces Pods. Gradually replaces old pods with new ones. Zero downtime.




#### Advantages: 
- Zero downtime
- Built into Kubernetes
- Safe
- Easy rollback

#### Disadvantages:
- Two versions run simultaneously
- Database schema changes can be challenging


#### Best Use Cases: 
- Most production applications





#### Workflow:







#### Deployment YAML:

```
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxUnavailable: 1
    maxSurge: 1              # Extra pods above desired count
```


#### Example: 

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: deploy-rolling
  labels:
    type: deploy-rolling

spec:
  replicas: 5
  minReadySeconds: 5
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1           # creates a new Pod first (because maxSurge: 1). 1 extra pod above desired (5+1=6 pods max) 
      maxUnavailable: 0     # Or 1 (0 means: No pod goes down before a new one is Ready). Guarantees at least 4 pods (5 - 1) must remain online

  selector:
    matchLabels:
      app: deploy-rolling

  template:
    metadata:
      labels:
        app: deploy-rolling

    spec:
      containers:
      - image: kodekloud/webapp-color:v1
      #- image: kodekloud/webapp-color:v2
        name: deploy-rolling-ct 
        ports:
        - containerPort: 8080
```



```
kubectl apply -f deploy-rolling.yaml
```



```
kubectl describe deployment deploy-rolling | grep Image
```



```
kubectl port-forward --address=0.0.0.0 deployments/deploy-rolling 80:8080
```



_Check rollout history:_
```
kubectl rollout history deployment/deploy-rolling
```



_If the new version has issues then rollback to a specific revision:_
```
kubectl rollout undo deployment/deploy-rolling --to-revision=1
```



_Monitor the rollback:_
```
kubectl rollout status deployment/deploy-rolling
```



`Note`: Without a defined `readinessProbe`, Kubernetes considers a container "healthy" the absolute millisecond its process starts up. **Always use readiness probes with rolling updates.**

### 3. Blue-Green Deployment:

Two complete environments exist simultaneously. Run two identical environments. Switch traffic all at once via a Service selector change.

- Blue = Current production version
- Green = New version


#### Advantages: 
- Instant rollback
- Near-zero downtime
- Easy testing before switching traffic


#### Disadvantages:
- Double infrastructure cost
- Requires twice the resources


#### Best Use Cases: 
- Banking
- E-commerce
- Enterprise production systems



#### Workflow:



#### Deployment YAML:

```
# Green (new) deployment
metadata:
  labels:
    version: green


# Switch the Service to green
spec:
  selector:
    version: green   # blue to green
```


### Example: 

`Important`: Kubernetes does **not provide Blue-Green deployments natively** through the `Deployment` resource. Instead, you implement them by running two Deployments and switching the Service selector (or by using tools like Argo Rollouts, Istio, or Linkerd).


#### Step 1: Deploy the Blue Version (currently live)


```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: app-blue
  labels:
    app: my-app
    slot: blue
    version: "1.0.0"

spec:
  replicas: 5

  selector:
    matchLabels:
      app: my-app
      slot: blue

  template:
    metadata:
      labels:
        app: my-app
        slot: blue

    spec:
      containers:
      - name: my-app-ct
        image: kodekloud/webapp-color:v1
        ports:
        - containerPort: 8080
```

```yaml
    slot: blue
```

```yaml

  selector:
    app: my-app
    slot: green
```

### 4. Canary Deployment:

A Canary Deployment gradually rolls out a new application version to a **small percentage of users** before exposing it to everyone.


Instead of replacing all Pods, you run two Deployments:

- **Stable Deployment (v1)** → Receives most traffic.
- **Canary Deployment (v2)** → Receives a small percentage of traffic.


#### Advantages: 
- Low risk
- Easy monitoring
- Detect issues before a full rollout


#### Disadvantages:
- More complex
- Requires monitoring and observability


#### Best Use Cases: 
- Large production systems
- SaaS platforms


#### Architecture:

```
                    Users
                      │
                      ▼
              +---------------+
              |    Service    |
              +---------------+
                      │
          -------------------------
          │                       │
          ▼                       ▼

   Stable Deployment        Canary Deployment
      nginx:v1                nginx:v2

      9 Pods                   1 Pod

Approximate Traffic:
      90%                      10%
```




#### Deployment YAML:

```
# Deployment-v1 : 9 replicas
# Deployment-v2 : 1 replica  →  ~10% of traffic
```


```
# Istio VirtualService
- destination:
    host: my-app
    subset: stable
  weight: 90
- destination:
    host: my-app
    subset: canary
  weight: 10
```




### Example: 


#### Step 1: Deploy the Stable Version (v1) - (90% of traffic)


```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: app-stable
  labels:
    app: my-app
    track: stable
    version: "1.0.0"

spec:
  replicas: 9

  selector:
    matchLabels:
      app: my-app
      track: stable    

  template:
    metadata:
      labels:
        app: my-app       # ← shared "app: my-app" label routes traffic
        track: stable

    spec:
      containers:
```

_If the canary is healthy, increase its replicas:_
```
-- Traffic ≈75% Stable - ≈25% Canary
kubectl scale deployment app-canary --replicas=3

-- Traffic ≈64% Stable - ≈36% Canary
kubectl scale deployment app-canary --replicas=5

-- Traffic ≈50% Stable - ≈50% Canary
kubectl scale deployment app-canary --replicas=9


-- Finally:
kubectl scale deployment app-canary --replicas=10
kubectl scale deployment app-stable --replicas=0

```



#### Step 5: Roll Back

If problems are detected:

```
-- Kill canary immediately — 100% traffic snaps back to stable
kubectl scale deployment app-canary --replicas=0 


-- Restore stable to full replica count
kubectl scale deployment app-stable --replicas=10 


-- Check deployment status
kubectl get deployment


-- Delete canary resources
kubectl delete deployment app-canary 
```

### 7. Rolling Back:

A rollback is the process of reverting an application to a previous working version when a deployment introduces bugs, failures, or performance issues.


```
-- Check rollout status / Watch rollout progress
kubectl rollout status deployment/deploy-rolling


-- Check rollout history
kubectl rollout history deployment/deploy-rolling
kubectl rollout history deployment/deploy-rolling --revision=1


-- Before or after updating the deployment, add the annotation
kubectl annotate deployment deploy-rolling kubernetes.io/change-cause="Updated image from v1 to v2" --overwrite
kubectl describe deployment deploy-rolling


-- Rollback to previous version
kubectl rollout undo deployment/deploy-rolling


-- Rollback to a specific revision
kubectl rollout undo deployment/deploy-rolling --to-revision=1
```
